Loom Presentation Script: A to Z

Exactly what to say, what to show on your screen, and how to look like a pro while explaining your POC to your boss.

1. Preparation & Setup

Before you hit record on Loom, get your screen set up so you aren't fumbling with windows during the recording.

Pre-recording Checklist:
  • Terminal: Start Redis (redis-server) and MongoDB (mongod).
  • Terminal: Start the app using node server.js in the backend/ folder. Keep this terminal visible or easily accessible so you can show the console logs.
  • Browser Tab 1: Open http://localhost:3000.
  • Browser Tab 2: Have the PROVIDER_DECISION.md file open in GitHub (or your IDE) to show the cost table.
  • IDE (VS Code): Open auth.js and otp.js so you can quickly tab to them.

2. The Script (Step-by-Step)

Part 1: The Hook & Introduction

(Screen showing: The main UI at localhost:3000)

"Hi [Boss's Name], I've finished the Mobile Number Authentication POC. Today I’m going to walk you through the live demo, explain why I chose our SMS/WhatsApp provider, and briefly show you the architecture and security measures I built into the code."

Part 2: The Live Demo

(Action: Type a 10-digit number like 9876543210 into the UI and hit Send OTP)

"Let's start with a demo. I enter a mobile number here. When I click send, the backend generates a secure OTP and fires it to both SMS and WhatsApp simultaneously.

Since I'm running this locally in mock mode without real API keys, the OTP is displayed right here in this blue banner for testing. If this were production, the OTP would obviously only go to the user's phone."

(Action: Type the OTP into the boxes)

"I'll enter the OTP. The system verifies it against Redis. Because this is a brand new phone number, it routes me to the registration page."

(Action: Fill out the name and email, and hit Register)

"I'll enter some basic details. Once I register, my account is saved to MongoDB, and I'm dropped into the dashboard where I can see my profile.

If I log out and use the exact same number again, the system will recognize me and skip registration, taking me straight to this dashboard."

Part 3: The Business Decision (Why MSG91)

(Action: Switch your screen to PROVIDER_DECISION.md, highlighting the cost table)

"One of the biggest requirements was finding the cheapest reliable provider and strictly avoiding Twilio due to cost. I evaluated 7 different providers.

I chose MSG91. Here's why:
1. It is 60–70% cheaper than Twilio.
2. It supports dual-channel delivery, so we hit SMS and WhatsApp with a single API call.
3. They pass through Meta's exact WhatsApp rate of 12 paise with zero markup.

At a volume of 50,000 OTPs a month, MSG91 will cost us roughly ₹13,000 to ₹16,000 total. Twilio would easily cost us over ₹45,000 for that same volume. So this choice saves us a lot of money while keeping delivery highly reliable via Indian telcos."

Part 4: The Code & Architecture

(Action: Switch your screen to VS Code, showing server.js or the folder structure)

"On the technical side, the stack is Node.js and Express, with MongoDB for permanent user data, and Redis for transient OTP data.

I structured the app using an MVC pattern. We have our Routes, Services, and Models separated. (Briefly click or point to the folder tree on the left)."

(Action: Open otp.js)

"I want to highlight why I used Redis here. Redis has a feature called TTL (Time-To-Live). When I save an OTP in Redis, I tell it to expire in exactly 5 minutes. Redis automatically deletes it from memory after 5 minutes, which means we don't have to write any messy cleanup scripts or background cron jobs. It's perfectly suited for OTPs."

Part 5: Security Measures

(Action: Still in otp.js, point to the generateOtp function and verifyOtp function)

"I took security very seriously in this POC. A few key measures:
1. OTPs are generated using crypto.randomBytes, not Math.random, so they are cryptographically secure and unpredictable.
2. To prevent brute-force attacks, the system tracks failed attempts. If a user guesses wrong 3 times, their number is completely blocked for 15 minutes.
3. We also have a 60-second cooldown on resending OTPs to prevent SMS bombing."

(Action: Open auth.js and scroll to the /register route)

"Finally, on the registration route, there is a strict Verification Gate. You cannot just ping the register API and create fake accounts. The system checks Redis for a 'verified' flag that is only granted if you successfully typed in the correct OTP. No OTP, no account creation."

Part 6: Testing & Wrap-Up

(Action: Open your terminal and run npm test)

"To ensure everything is bulletproof, I wrote a full suite of automated integration tests. (Let the tests run and show the green passing marks).

As you can see, all 26 tests are passing. They test the entire OTP lifecycle, the rate limiting, and all the auth routes without needing to hit the real MSG91 API.

That’s the A to Z of the project! Let me know if you want to dive deeper into any of the code, or if you're ready for me to push this forward. Thanks!"

3. Pro-Tips for a Great Loom

  • Keep it under 5 minutes. Bosses are busy. The script above should take exactly 3.5 to 4.5 minutes to read at a normal pace.
  • Don't read it like a robot. Use the script as a bulleted guide. It's okay if you paraphrase.
  • Practice once before recording. Do a dry run where you just click between the windows so you know exactly where your mouse needs to go.
  • If you make a small mistake, keep going. Loom videos don't need to be Hollywood productions. If you say "Mongo" instead of "Redis" by accident, just correct yourself and keep talking.
