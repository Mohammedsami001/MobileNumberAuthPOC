1. The Big Picture — What This App Does

This is a mobile number authentication service. Instead of passwords, users prove who they are by receiving a one-time password (OTP) on their phone — both via SMS and WhatsApp simultaneously.

Why OTP instead of passwords?
No password database to leak. No "forgot password" flow. The user's phone IS the credential. This is standard for Indian apps — Swiggy, Zomato, PhonePe, Paytm all work this way.

The complete user flow has 4 steps:

1
Enter mobile number User types their 10-digit Indian mobile number. The app validates it (must start with 6-9, exactly 10 digits).
2
Send OTP Backend generates a secure 6-digit code, stores it in Redis (auto-expires in 5 min), and fires SMS + WhatsApp simultaneously through MSG91.
3
Verify OTP User enters the 6-digit code. Backend checks it against Redis. Wrong code? Track attempts — block after 3 failures for 15 minutes. Right code? Proceed.
4
Route the user Check MongoDB — does this mobile number already have an account?
• New user → show registration form (name, email, city)
• Existing user → go straight to dashboard with their profile

What is Redis TTL?

When you store a value in Redis, you can say "delete this automatically after X seconds". This is called TTL (Time To Live).

// "Store this OTP, but delete it after 300 seconds (5 minutes)"
await redis.set("otp:9876543210", "482931", "EX", 300);

// After 5 minutes, this returns null — the OTP expired
await redis.get("otp:9876543210"); // → null

This is why Redis is perfect for OTPs. You don't need a background job checking "has this OTP expired?" — Redis handles it automatically.

Why is the code split into routes/, services/, and models/?
This is the MVC-like pattern. Routes handle HTTP (what the user sends and receives). Services handle business logic (OTP rules, rate limiting). Models handle data shape (what a User looks like in the database). Each layer only talks to the one below it — routes call services, services call models/Redis.

4. The 4 API Endpoints — Step by Step

Endpoint 1: POST /auth/send-otp

What it does: Takes a mobile number, generates an OTP, stores it in Redis, sends it via SMS + WhatsApp.

// Frontend sends:
{ "mobile": "9876543210" }

// Backend does:
1. Validate: starts with 6-9, exactly 10 digits?  → 400 if no
2. Check Redis: is this number blocked?            → 429 "too many attempts"
3. Check Redis: was OTP sent less than 60s ago?    → 429 "wait before resending"
4. Generate OTP: crypto.randomBytes → 6-digit code
5. Store in Redis: otp:9876543210 = "482931" (TTL 5 min)
6. Set cooldown: otp_cooldown:9876543210 = "1" (TTL 60 sec)
7. Call MSG91: fire SMS and WhatsApp simultaneously
8. Return: { success: true, message: "OTP sent" }

Endpoint 2: POST /auth/verify-otp

What it does: Checks the submitted OTP against Redis. If correct, checks if the user exists in MongoDB.

// Frontend sends:
{ "mobile": "9876543210", "otp": "482931" }

// Backend does:
1. Get stored OTP from Redis: otp:9876543210
2. If no key → OTP expired (TTL ran out)                    → 401 "EXPIRED"
3. Compare using crypto.timingSafeEqual (constant-time)
4. If wrong → increment attempts, check if ≥ 3              → 401 "INVALID" or "MAX_ATTEMPTS"
5. If correct:
   a. Delete OTP + attempt keys from Redis (cleanup)
   b. Set verified:9876543210 = "1" (TTL 10 min) in Redis
   c. Check MongoDB: User.findOne({ mobile: "9876543210" })
   d. Return { isNewUser: true } or { isNewUser: false, profile: {...} }
What is timingSafeEqual and why does it matter?
Normal string comparison (===) returns false as soon as the first wrong character is found. An attacker can measure how long the comparison takes to guess characters one by one. timingSafeEqual always takes the same time regardless of where the mismatch is, preventing this timing attack.

Endpoint 3: POST /auth/register

What it does: Creates a new user account. Only works if the mobile number was OTP-verified.

// Frontend sends:
{ "mobile": "9876543210", "name": "Priya Sharma", "email": "...", "city": "Mumbai" }

// Backend does:
1. Validate mobile number and name
2. CHECK REDIS: is verified:9876543210 set?      → 403 if not!
3. CHECK MONGODB: does user already exist?        → 409 if yes
4. Create new User in MongoDB
5. Delete verified key (one-time use)
6. Return: { success: true, profile: {...} }
The Verification Gate — this is critical.
Without step 2, anyone could call POST /auth/register directly with any phone number and create a fake account. The verified:{mobile} Redis key ensures that only someone who actually received and typed the correct OTP can register. This key auto-expires after 10 minutes.

Endpoint 4: GET /auth/profile/:mobile

What it does: Fetches user profile from MongoDB. Used for the dashboard screen.

// Browser requests:
GET /auth/profile/9876543210

// Returns:
{ name, mobile, email, city, memberSince, lastLogin, loginCount }
Boss question: "Is the profile endpoint secure?"
For the POC, no — it's unauthenticated. In production, you'd add JWT tokens: after OTP verification, the server issues a signed token, and the profile endpoint requires that token in the Authorization header. This was intentionally scoped out of the POC.

5. The OTP Service — Deep Dive into otp.js

OTP Generation: Why crypto.randomBytes?

// ❌ BAD — Math.random() is predictable
const otp = Math.floor(100000 + Math.random() * 900000);

// ✅ GOOD — crypto.randomBytes is cryptographically secure
const bytes = crypto.randomBytes(4);      // 4 random bytes from OS
const num = bytes.readUInt32BE(0);         // Convert to integer
const otp = String(100000 + (num % 900000)); // Map to 6-digit range

Math.random() uses a pseudorandom number generator (PRNG) — if you know the seed, you can predict every output. crypto.randomBytes() uses the operating system's secure entropy source (/dev/urandom on Linux, CryptGenRandom on Windows). It's unpredictable.

Rate Limiting: The 3-Layer Defense

LayerWhat it preventsHow
Cooldown (60s)OTP spam / SMS bombingAfter sending OTP, set a 60s Redis key. If it exists, refuse to send another.
Attempt limit (3 tries)Brute-force guessingTrack wrong guesses. After 3 wrong OTPs → block.
Block (15 min)Persistent attackersAfter hitting max attempts, set a 15-min block key. No OTP send or verify allowed.

7. MSG91 Integration — How the Code Works

The msg91.js service has a built-in mock mode. If MSG91_API_KEY is not set, it doesn't call the real API — it just prints the OTP to the console.

// Mock mode (no API key set):
console.log("[MOCK SMS] → 9876543210: OTP is 482931");
// Returns: { success: true, mock: true }

// Real mode (API key configured):
// Makes HTTP POST to MSG91 API
// Returns: { success: true, mock: false }

Both channels fire simultaneously using Promise.allSettled:

const [smsResult, waResult] = await Promise.allSettled([
  sendSmsOtp(mobile, otp),      // fires SMS
  sendWhatsAppOtp(mobile, otp)  // fires WhatsApp at the same time
]);

Why allSettled instead of Promise.all? Because Promise.all would fail BOTH if either one fails. allSettled lets SMS succeed even if WhatsApp fails (or vice versa). Success = at least one channel delivered.

Mock-mode safety guard:
In dev mode, the API response includes _devOtp: "482931" so you can test without a phone. But if NODE_ENV=production, this field is NEVER sent — even if the MSG91 key is accidentally unset. This prevents leaking OTPs in production.

11. server.js — The Entry Point

The server file does 5 things:

  1. Loads config from .env (uses path.join(__dirname, "../.env") because the .env is in the project root but server.js is in backend/)
  2. Sets up middleware: JSON body parsing, CORS, static file serving for the frontend
  3. Connects to MongoDB via Mongoose
  4. Mounts routes: all auth routes live under /auth/*
  5. Error middleware: catches unhandled errors and returns { success: false, message: "Internal server error." } instead of leaking stack traces

It also exports the app object for testing:

// This lets supertest import the app without starting the server

// The server only boots when run directly (not when imported by tests)
if (require.main === module) {
  start();
}

14. Self-Check — Do You Really Know This?

Try to answer each question before clicking the answer. If you can answer all of these from memory, you're ready.

What Node.js function generates the OTP, and why not Math.random()?

  • Math.random() — it's fast and good enough for 6 digits
  • crypto.randomBytes() — it's cryptographically secure and unpredictable
  • uuid.v4() — it generates unique identifiers
  • Date.now() % 1000000 — it uses the current timestamp
crypto.randomBytes() pulls entropy from the OS's secure random source. Math.random() is a PRNG — predictable if you know the seed.

What does Redis TTL do for OTPs in this project?

  • Logs the OTP creation time for auditing purposes
  • Encrypts the OTP with a time-based key rotation
  • Automatically deletes the OTP after 5 minutes — no cleanup code needed
  • Sends a notification when the OTP is about to expire
TTL = Time To Live. Redis deletes the key automatically when the TTL expires. The OTP simply ceases to exist after 5 minutes.

Why does the /register endpoint check for a "verified:{mobile}" key in Redis?

  • To check if the user's email has been confirmed
  • To ensure the mobile number was OTP-verified before allowing registration
  • To verify the user's identity against a government database
  • To check if the user has completed a CAPTCHA challenge
Without this gate, anyone could call POST /register directly with any phone number and create a fake account. The verified key proves they received and entered the correct OTP.

Why does the app use Promise.allSettled instead of Promise.all for SMS + WhatsApp?

  • Promise.allSettled is newer and therefore faster
  • Promise.all doesn't support multiple async functions
  • Promise.allSettled retries failed requests automatically
  • Promise.all fails everything if one fails — allSettled lets SMS succeed even if WhatsApp fails
If WhatsApp delivery fails but SMS succeeds, the OTP was still delivered. Promise.all would reject both results if either threw an error. allSettled gives you the result of each independently.
